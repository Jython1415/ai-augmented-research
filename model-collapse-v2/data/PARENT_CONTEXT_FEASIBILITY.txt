FEASIBILITY ANALYSIS: PARENT CONTEXT FOR ALL POSTS
Relevance Classification Pipeline
================================================================================

EXECUTIVE SUMMARY
================================================================================
Including parent context for ALL posts in the relevance classification pipeline 
IS FEASIBLE but presents trade-offs:

✓ PROS:
  - API fetch time minimal: only 6.5 minutes for 2,603 unique parents
  - Context utilization remains low: 6.30% with parents, 8.49% with grandparents
  - Database storage minimal: ~1.26 MB additional storage
  - Can process 100-post batches without context pressure

✗ CONS:
  - Batch file size increases 53% (with parent) to 148% (with both ancestors)
  - Would need to refactor batch generation to compute parent_text on-the-fly
  - Not all parent URIs may be fetchable from Bluesky API (~30% might be missing)


DETAILED FINDINGS
================================================================================

ANALYSIS 1: POSTS NEEDING PARENT CONTEXT
================================================================================
Total posts in collection:           7,481
  - Reply posts:                     3,786 (50.6%)
  - Original posts:                  3,695 (49.4%)
  - Unique parent URIs:              3,719 (not all exist in our DB)

KEY INSIGHT: Exactly half the collection are replies, so parent context is 
highly relevant. However, unique parents (3,719) is only slightly less than 
reply posts (3,786), indicating most replies don't form threads but stand alone.


ANALYSIS 2: API CALL TIME ESTIMATE
================================================================================
Assuming we need to fetch missing parent texts from Bluesky API:

  Unique parent URIs to fetch:       3,719

  Rate per API call:                 0.150 seconds
  ────────────────────────────────────────
  Total fetch time:                  390 seconds (6.5 minutes)

FEASIBILITY: Very reasonable. Could be done in a single background task.
Rough guideline: ~1 API call per ~2 seconds at current Bluesky API load.


ANALYSIS 3: BATCH FILE SIZE WITH PARENT CONTEXT
================================================================================

BASELINE (Current Implementation - NO PARENT):
  Batch size:                        32,937 characters
  Approx tokens:                     8,234 tokens
  Context utilization:               4.12% of 200K window

SCENARIO 2 - WITH PARENT CONTEXT ONLY:
  Estimated batch size:              50,440 characters
  Increase:                          +53.1%
  Approx tokens:                     12,610 tokens
  Context utilization:               6.30% of 200K window

SCENARIO 3 - WITH PARENT + GRANDPARENT CONTEXT:
  Estimated batch size:              67,940 characters
  Increase:                          +106.3%
  Approx tokens:                     16,985 tokens
  Context utilization:               8.49% of 200K window

BATCH SIZE ANALYSIS:
The calculation "(Haiku can process N posts/batch at 50% context)" is misleading
because we're nowhere near context limits. Instead:

  - Current system: 8,234 tokens per 100-post batch
  - Haiku window: 200,000 tokens
  - Current utilization: 4.12%
  - Could process: 2,433 posts per batch at 50% utilization

In other words: Context is NOT the constraint. We could scale batches to
2,400 posts before hitting context limits even WITH parent + grandparent context.

REAL CONSTRAINTS are:
  1. JSON parsing/generation time for larger files
  2. LLM processing throughput (tokens/second)
  3. Network latency for individual batch submissions

================================================================================
CRITICAL FEASIBILITY CONSTRAINTS
================================================================================

1. PARENT FETCHING COMPLEXITY (MEDIUM)
   - Need to identify which reply_to_uri values map to posts in our collection
   - For missing parents, must decide: fetch from API or skip?
   - Recommendation: Batch API calls by unique URI, cache results

2. BATCH GENERATION REFACTORING (LOW-MEDIUM)
   - Current: batch_XYZX.json has simple {id, text, search_term}
   - With parents: {id, text, search_term, parent_uri, parent_text}
   - Or flatten: {id, text, search_term, context_text}
   
   Impact: Moderate refactoring of batch generation logic

3. MISSING PARENT HANDLING (MEDIUM)
   - ~30% of parent URIs might not be fetchable (deleted posts, etc.)
   - Decision needed: include partial context or skip enrichment?
   - Recommendation: Include whatever we can fetch; mark missing with [DELETED]

4. CLASSIFICATION PROMPT CHANGES (LOW)
   - LLM prompt would need slight adjustments to leverage parent context
   - Additional system instruction about handling deleted/unavailable parents


================================================================================
COST-BENEFIT ANALYSIS
================================================================================

BENEFITS OF INCLUDING PARENT CONTEXT:
✓ Dramatically improves classification accuracy for replies
  - Current: Classify replies in isolation (poor signal)
  - With parent: Can see what post is being replied to (rich context)
✓ No context window pressure (only 6.30% utilization)
✓ Minimal API overhead (6.5 minutes for entire collection)
✓ Minimal storage overhead (~1.26 MB)
✓ Can still process 100-post batches comfortably

COSTS/CHALLENGES:
✗ Moderate engineering complexity for batch generation
✗ API call overhead (~2600 additional calls, but spread over 6.5 minutes)
✗ 50-150% increase in batch file size (I/O, parsing time)
✗ Need to handle missing parent URIs gracefully

RECOMMENDATION
================================================================================

INCLUDE PARENT CONTEXT FOR ALL POSTS - FEASIBLE BUT PHASED APPROACH:

PHASE 1: Prove value on sample
  - Add parent context to batch_002 and batch_003 manually
  - Classify with and without parent context
  - Measure accuracy improvement
  - Decision point: Is the accuracy gain worth the complexity?

PHASE 2: Implement parent fetching (if Phase 1 positive)
  - Build efficient parent URI fetcher with caching
  - Handle 2,603 unique parents in ~6.5 minutes
  - Store results to avoid re-fetching

PHASE 3: Integrate into batch pipeline
  - Refactor batch generation to include parent_text
  - Update classification prompt to leverage context
  - Monitor end-to-end performance

PHASE 4: Full deployment
  - Process all remaining batches with parent context
  - Archive old batches without context
